<task type="auto">
  <name>Task 2: Build recommendation cards UI on the interview page</name>
  <files>apps/mesh/src/web/routes/onboard-interview.tsx</files>
  <action>
Extend the interview page (created in plan 01) to render a recommendations view after the interview completes.

**The interview page has two view states**, controlled by the `step` search param:
- No step / step=interview: The chat interview (plan 01 built this)
- step=recommendations: The recommendation cards (this task builds this)

**When step=recommendations:**

1. **Fetch recommendations** using `useQuery`:
   ```typescript
   const { data: recommendations, isLoading } = useQuery({
     queryKey: KEYS.onboardingRecommendations(token ?? ""),
     queryFn: () =>
       fetch(`/api/onboarding/recommendations?token=${encodeURIComponent(token ?? "")}&organizationId=${encodeURIComponent(organizationId ?? "")}`)
         .then(r => {
           if (!r.ok) throw new Error(`HTTP ${r.status}`);
           return r.json();
         })
         .then(data => data.recommendations as AgentRecommendation[]),
     enabled: !!token && !!org,
   });
   ```

   The `organizationId` needs to come from somewhere. Options:
   - Fetch it via the session or the resolve endpoint. Simplest: the interview-results endpoint (plan 01) can return the organizationId in its response, and we pass it as a search param to step=recommendations.
   - Or: add `organizationId` to the recommendations search params.

   Use the approach that fits best with the flow from plan 01 — when the interview completes and navigates to step=recommendations, include the organizationId as a search param.

2. **Render recommendation cards** — each card is an `AgentRecommendationCard` component:

   Layout for each card:
   ```
   ┌─────────────────────────────────────────────┐
   │  [Icon]  Agent Title                         │
   │  Agent description (1-2 lines)               │
   │                                              │
   │  "Recommended because..."                    │
   │  [reason text — personalized explanation]     │
   │                                              │
   │  Connections needed:                          │
   │  ● Connection A  ✓ Connected                 │
   │  ● Connection B  → Connect                   │
   │                                              │
   │  [         Hire this agent         ]          │
   └─────────────────────────────────────────────┘
   ```

   Styling:
   - Card: `rounded-xl border border-border bg-card p-6 shadow-sm` (same as onboard-setup cards)
   - Agent icon: Use `IntegrationIcon` component if available, fallback to a generic agent icon (Users03 from untitledui)
   - Title: `text-base font-semibold text-foreground`
   - Description: `text-sm text-muted-foreground mt-1`
   - Reason: `text-sm text-foreground mt-3` with a subtle accent — prefix with a lightbulb or sparkle icon
   - Connection list: `mt-3 space-y-1.5` — each connection as a row with status badge
   - Connected status: green dot + "Connected" text
   - Not connected: amber dot + "Connect" as a clickable link
   - "Hire this agent" button: `rounded-lg bg-brand px-4 py-2.5 text-sm font-semibold text-brand-foreground` (same brand button as claim flow)

3. **Connect action (AGNT-03, AGNT-04):**

   Clicking "Connect" on a not-connected connection item OR clicking "Hire this agent" should:
   - Navigate to `/${org}/org-admin/connections` with search params that trigger the add connection dialog with pre-populated type
   - The connections page reads search params `?add=true` to open the "Add Connection" dialog automatically
   - To pre-populate the connection type: check how the connections page form works. It uses a `ui_type` select (HTTP, SSE, Websocket, NPX, STDIO). The connection from the Virtual MCP has a `connection_id` — look up the actual connection entity to find its type.

   Implementation: Use `window.location.href` to navigate (since we're outside the shell layout, using the router would not work correctly for cross-layout navigation):
   ```typescript
   window.location.href = `/${org}/org-admin/connections?add=true`;
   ```

   If the connection already has enough info to pre-populate (like a known URL or type), pass it: `?add=true&url=<connectionUrl>&type=<SSE|HTTP>`.

4. **Empty state:** If `recommendations` is an empty array:
   - Show a friendly message: "No agent recommendations yet — your team doesn't have any agents configured. You can add agents later from the dashboard."
   - "Go to dashboard" button: `window.location.href = \`/\${org}\``

5. **Loading state:** While recommendations are loading, show skeleton cards (3 placeholder cards with `animate-pulse` similar to SetupSkeleton pattern).

6. **Page layout for recommendations view:**
   - Full-screen centered layout (same as interview/setup pages)
   - Header: "Recommended agents for your store" with the storefront URL below
   - Subheading: "Based on your diagnostic results and goals, we recommend hiring these agents:"
   - Cards in a vertical stack with `space-y-4`
   - Footer: "Skip for now" link that goes to `/${org}` — always visible, users shouldn't feel trapped

7. **Navigation from interview to recommendations:**
   Update the interview completion handler (from plan 01) to navigate to the recommendations view. After the interview-results POST succeeds, set `window.location.href = \`/onboard-interview?org=\${org}&token=\${token}&step=recommendations\`` (or use TanStack Router navigate with updated search params if within the same route).
